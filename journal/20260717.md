# 2026-07-17 — First Hardware Capture, CSI Debugging & Subcarrier Fix

## Summary

Ran the first real capture with two Arduino-flashed ESP32 anchors. Diagnosed
and cleared a "no CSI" result (cause was channel/traffic, **not** a disabled
build) and a serial "Resource busy" error (Arduino Serial Monitor holding the
ports). Confirmed passive multi-device sniffing with real spatial diversity.
The real data resolved the long-standing `first_word_invalid` question:
subcarrier +1 is corrupt, so `gateway/csi.py` now drops it by default (51
usable subcarriers → 50 D-CFR), with a flag to restore the textbook 52/51.
Full pipeline validated on the real capture; 45 tests passing.

## Log

- Confirmed `gateway_logger.py` works with the Arduino firmware (identical line
  format to the ESP-IDF build; the two can even be mixed across anchors).
- First single-board test emitted **only** `STAT` lines, e.g.
  `STAT,A1,20101,200,0,0,0,215468` → decoded as uptime 20 s, `pkts_seen=200`,
  **`csi_cb_count=0`**. Conclusion: promiscuous sniffing works, but the CSI
  path produced nothing.
- Formed two hypotheses: (1) CSI compiled out of the Arduino core (the sketch
  ignored the `esp_wifi_set_csi*` return codes, so a failure would be silent);
  (2) no OFDM frames on the sniffed channel / wrong channel.
- Clarified the passive-sniffing premise: the real system joins no network;
  CSI comes from the **unencrypted L-LTF preamble**, so no network access is
  needed. Requirements are only (a) anchor parked on the target's channel and
  (b) the target actively transmitting OFDM frames.
- Noted two practical constraints: the classic ESP32 is **2.4 GHz only** (a
  phone on 5 GHz is invisible), and modern phones use **randomized MACs** so
  the sniffed MAC won't match the target's real one unless randomization is
  disabled.
- Set `WIFI_CHANNEL` to 4 and ran the logger on two boards → both ports failed
  with `Errno 16 Resource busy`.
- Diagnosed with `lsof`: two Arduino IDE `serial-monitor` helper processes were
  holding `/dev/cu.usbserial-0001` (A1) and `/dev/cu.usbserial-4` (A2). A port
  can only be held by one program at a time.
- Closed the Serial Monitors and re-ran the logger → real data captured:
  **A1 = 486 CSI + 19 STAT, A2 = 462 CSI + 19 STAT**. CSI confirmed working —
  the earlier zero was cause (2), no OFDM traffic on the channel.
- Ran the gateway parser over both files: **948 of 949 CSI lines clean**, 1
  malformed (a mangled first line from the logger connecting mid-stream and
  reading a backlog — harmless; the parser counts and skips it).
- Captured **three devices passively** with clear spatial diversity, e.g.
  `fe:e3:be:47:e1:f0` reads −45.9 dBm at A1 vs −29.2 dBm at A2 (A2 closer).
  `fe:..` and `46:..` are randomized MACs; `98:0d:af:75:12:28` is a real vendor
  OUI (a nearby real device, ~−14 dBm).
- Checked the subcarrier layout on real data (mean amplitude per hardware
  index): index 0 (DC) = 95.7 (already dropped), **index 1 (subcarrier +1) =
  3.4 — anomalous vs neighbors ~25**, guard indices 27–37 = 0 (already
  dropped). This confirmed subcarrier +1 is corrupt (`first_word_invalid`).
- Applied the fix: added a `drop_first_pos` config to `gateway/csi.py`,
  **defaulting to drop** subcarrier +1 → 51 usable / 50 D-CFR; threaded it
  through `features`/`preprocess`; added `--keep-first-subcarrier` to restore
  52/51; recorded the choice in the output `_meta`.
- Updated the test suite (now **45 passing**) and corrected the subcarrier note
  in `firmware/README.md`.
- Ran the full pipeline on the real capture: **120 windows, 105 sufficient
  (87.5% usable rate)** at `--min-anchors 2`, D-CFR vectors produced; verified
  both default (50-D-CFR) and `--keep-first-subcarrier` (51-D-CFR) modes.

## Issues & Fixes

1. **No CSI captured (`csi_cb_count=0`).** Cause: the sniffed channel had no
   OFDM traffic (and/or the target was elsewhere). Fix: set `WIFI_CHANNEL` to a
   channel with an active OFDM transmitter — CSI then flowed immediately. (The
   Arduino core does have CSI enabled; hypothesis (1) was ruled out.)
2. **`Errno 16 Resource busy` on both serial ports.** Cause: Arduino IDE Serial
   Monitor held the ports. Fix: close all Serial Monitors before running
   `gateway_logger.py` — a port allows only one reader.
3. **Corrupt subcarrier +1 (`first_word_invalid`).** Cause: the ESP32 marks the
   first CSI word (hardware indices 0–1) invalid; index 1 is subcarrier +1 and
   reads ~3 vs ~25 for neighbors. Fix: drop it by default in `gateway/csi.py`
   (config-flag, `--keep-first-subcarrier` reverts).
4. **One mangled first line per file.** Cause: logger connecting mid-stream and
   reading a buffered backlog. Impact: negligible (parser skips + counts it).
   Optional future fix: flush the serial input buffer on connect.

## Setup / Reproduce

- **Flash (per board):** Arduino IDE → board "ESP32 Dev Module" → set
  `ANCHOR_ID` (A1, A2, …) and `WIFI_CHANNEL` to the target's 2.4 GHz channel →
  Upload. Change `ANCHOR_ID` and the selected Port between boards.
- **Free the ports:** close every Arduino IDE Serial Monitor.
- **Capture:**
  `python firmware/tools/gateway_logger.py --port /dev/cu.usbserial-0001:A1 --port /dev/cu.usbserial-4:A2`
  → writes `data/A1_<ts>.csv`, `data/A2_<ts>.csv`. Stop with Ctrl-C.
- **Preprocess:**
  `.venv/bin/python -m gateway.preprocess --in data/ --out windows.jsonl --min-anchors 2 --summary`
  (add `--keep-first-subcarrier` for the textbook 52/51 layout).
